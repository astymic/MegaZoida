# Утвержденные правила генерации 3D-моделей и анимации (MegaZoida)

Этот документ строго регламентирует, как именно AI агенты должны генерировать 3D модели, ригинг и скрипты для Three.js, чтобы избежать багов с зависанием, скольжением и "парящими в воздухе" моделями.

## 1. Построение геометрии и центровка (Zero Hovering Rule)
Модели не должны собираться в единый ком с помощью `join()` *до* сброса `origin`. Это приводит к тому, что центр масс смещается вверх (в живот или грудь), и в игре модель начинает парить над землей (offset).
**Правильный пайплайн:**
1. Создаем отдельные примитивы (Голова, Тело, Рука_Л, Нога_П).
2. Поднимаем их вершины на нужную высоту с помощью параметров или режима редактирования. 
3. Делаем **Transform Apply**: `bpy.ops.object.transform_apply(location=True, rotation=True, scale=True)` для каждой части. Это гарантирует, что `origin` (точка отсчета) каждой части зафиксирована в `(0,0,0)`, а вершины подняты только локально!

## 2. Идеальный риггинг (Жесткие веса)
Никакого автоматического веса `Automatic Weights` для блочных/пиксельных персонажей. Из-за него рука будет оттягивать живот, а нога — сжимать таз.
**Правильный пайплайн:**
1. Вешаем на каждый Mesh-объект модификатор `Armature`.
2. Назначаем родителя `Parent = OBJECT` (к арматуре).
3. Создаем в Mesh одну-единственную группу вершин (`vertex_group`), имя которой СТРОГО совпадает с именем кости (например, `Arm_L`). Воруем все `verts` в эту группу с весом `1.0`. Никаких искажений.

## 3. Анимация и проблема FBX-схлопывания
Blender-FBXExporter имеет известный баг: он схлопывает несколько экшенов (`Idle`, `Walk`) в один `Take 001`, если они не добавлены правильно, из-за чего Three.js `AnimationMixer` начинает вычитать вес анимации из самой себя, и персонаж застывает в одной позе (обычно с выдвинутой ногой).
**Правильный пайплайн:**
1. Создаем `Action` для `Idle` и для `Walk`.
2. После расстановки ключей, **обязательно создаем NLA-треки** и запихиваем экшены в `strips`:
   ```python
   track = arm_obj.animation_data.nla_tracks.new()
   track.strips.new(action.name, start, action)
   ```
3. При экспорте FBX ставим флаги:
   `bake_anim_use_all_actions = True`
   `bake_anim_step = 1` (Обязательно, чтобы не терялись In-Betweens!)
   `bake_anim_simplify_factor = 0.0` (Запретить FBX портить ключи!)

## 4. Движок: Плавность и синхронизация ходьбы
В `Three.js` запрещено задавать `timeScale` грязными константами.
Нужно отвязать расстояние `STRIDE` (сколько пикселей/игровых метров герой проходит за 1 полный цикл анимации) от времени ролика.
**Правильная формула синхронизации стоп с землей:**
```typescript
// Считаем реальную длительность клипа, загруженного из FBX
const walkClipDuration = clip.duration; 
const STRIDE_LENGTH = 80; // Физическая длина шага

// В update():
actionWalk.timeScale = (speed / STRIDE_LENGTH) * walkClipDuration;
```
Это железобетонно фиксирует стопу на земле, позволяет изменять FPS в Blender, добавлять кадры, ускорять игру — ноги не будут скользить никогда.
